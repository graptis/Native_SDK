46 // Enable the following defines for improved validation of memory usage in debug builds
47 // VMA_DEBUG_INITIALIZE_ALLOCATIONS:
48 // Makes memory of all new allocations initialized to bit pattern `0xDCDCDCDC`.
49 // Before an allocation is destroyed, its memory is filled with bit pattern `0xEFEFEFEF`.
50 // If you find these values while debugging your program, good chances are that you incorrectly
51 // read Vulkan memory that is allocated but not initialized, or already freed, respectively.
52 #define VMA_DEBUG_INITIALIZE_ALLOCATIONS 1
53 // VMA_DEBUG_MARGIN:
54 // Enforces a specified number of bytes as a margin before and after every allocation.
55 // If your bug goes away after enabling margins, it means it may be caused by memory
56 // being overwritten outside of allocation boundaries.It is not100 % certain though. Change in application behavior may also be caused by different order and
57 // distribution of allocations across memory blocks after margins are applied.
58 #define VMA_DEBUG_MARGIN 4
59 // VMA_DEBUG_DETECT_CORRUPTION:
60 // If this feature is enabled, number of bytes specified as `VMA_DEBUG_MARGIN`
61 // (it must be multiply of 4) before and after every allocation is filled with a magic number. This idea is also know as "canary".
62 // Memory is automatically mapped and unmapped if necessary. This number is validated automatically when the allocation is destroyed.
63 // If it's not equal to the expected value, `VMA_ASSERT()` is executed. It clearly means that either CPU or GPU overwritten the
64 // memory outside of boundaries of the allocation, which indicates a serious bug.
65 #define VMA_DEBUG_DETECT_CORRUPTION 1
66 #endif
